Delphi offers a set of R packages for quickly making start-to-finish forecasting pipelines. The packages interface seamlessly with each other so that users can fetch and clean data, conduct custom data preprocessing, train a model, and make predictions all in one go.

These tools are meant for use with panel data, where a series of observations is reported over time for the same entity (e.g. a geographical area, or demographic cross-section, or both). Our tools are primarily focused on traditional time series models, such as autoregressive (AR) and AR with external predictors (ARX).

Notably, we provide utilities for handling datasets that have revisions. These utilities prevent information leakage when performing arbitrary calculations, including training models. epiprocess is the only package we’re aware of that does this natively and automatically. Correctly handling data revisions is extremely important for training and accurately measuring performance of new models on historical dates. Delphi’s tooling packages were built with this in mind.

In this post, we will implement a toy prediction model using the packages, evaluating them for ease of use, documentation quality, and prediction accuracy. Take a look at the post introducing this series for more background information.

Breaking down the tools

Delphi offers three R packages, epidatr, epiprocess, and epipredict. Each covers a different stage of the forecasting process.

epidatr is responsible for fetching data from the Delphi Epidata API, through which Delphi offers a variety of data. The API also stores historical versions of the data, as reported on different dates. This revision history is particularly useful for assessing the historical accuracy of new forecasting models (“backtesting”). We also have a python API client available, epidatpy.

epiprocess can clean and transform any panel data. Functions include simple hands-off plotting, calculating growth rate, outlier detection and removal, fast rolling computations with support for custom calculations, and more. Computations automatically handle revisions correctly.

Finally, epipredict is responsible for fitting time-series models and producing predictions for panel data. We support forecasting and in the near future are aiming to support nowcasting. Nowcasting learns patterns in the revision history to estimate not-yet-finalized indicator values. The package focuses on quick-build models, but users can also build fully custom models.

These three packages are meant to work together to fetch and manipulate epidemiological data, and create customizable, interpretable forecasts.

They are also designed to be easy to use out of the box with data sourced from the Delphi API, which makes the API convenient for quickly trying out Delphi tools. However, any panel data that can be read into R can also be used. We also provide a convenient function in epiprocess to aggregate line-level data into the correct format.

We are continuing to develop and streamline these tools for easier quick-start use and more intuitive debugging, and are continuing to add new features.

Now, on to the testcase.

Fetching data from the Epidata API

To start, we need test data. We used the function epidatr::pub_covidcast() (reference) to fetch NHSN hospitalization data from the Delphi Epidata API.

# Set up all the packages needed
pak::pkg_install("cmu-delphi/epiprocess")
pak::pkg_install("cmu-delphi/epipredict")

library(epiprocess)
library(epidatr)
library(epipredict)
library(data.table)
library(dplyr)
library(tidyr)
library(ggplot2)

# Fetch data from Delphi Epidata API
nhsn <- pub_covidcast(
  source = "nhsn",
  signals = "confirmed_admissions_covid_ew_prelim",
  geo_type = "state",
  time_type = "week",
  # Request data for all states.
  geo_values = "*",
  time_values = epirange(202001, 202530),
  # Request a particular set of data issues, not just the most recent version.
  issues = epirange(202401, 202530)
)  %>%
  # Do some reformatting for convenience
  rename(version = issue, confirmed_covid_adm = value) %>%
  select(geo_value, time_value, version, confirmed_covid_adm) %>%
  # Convert to `epi_archive` object
  as_epi_archive()

We end the code chunk converting our data frame into an epi_archive object, a special epiprocess class meant for storing panel data with version history. The conversion process validates the data and removes redundant rows.

Note: While the Delphi API can be used anonymously, it comes with limits on query complexity and amount of data fetched. We recommend using an API key to avoid these restrictions.

Delphi tools make working with data revisions easy

Many epidemiological datasets are subject to revision, where the value for a given date is modified over time as more information becomes available.

What sets Delphi apart from competitor forecasting tools is that our packages do all processing while keeping track of those revisions. Doing this correctly is difficult, and our tools save users from needing to write their own revision-handling code.

Using epiprocess to train a model

To generate forecasts, we used a simple autoregressive model with quantile regression, which produces a predicted distribution rather than a single point prediction. This can be done using the arx_forecaster() function. The function is designed to need minimal user input while still generating a reasonable forecast. epipredict also supports building fully custom models, but we won’t discuss that in this post.

We made two versions of the forecasting model to better exhibit the utility of the data revision-handling features. The version-faithful model properly considers the data versioning, only using values reported at the time a forecast was or would have been made. This is called backtesting. The version un-faithful model ignores the different data versions and pretends we knew the finalized version of the data all along, i.e. that the data had no revisions.

# Because `arx_forecaster` can only fit one forecast horizon ("ahead") at a time, create helper function to iterate over a list of several forecast horizons.
forecast_wrapper <- function(
    epi_data, aheads, outcome, predictors,
    # An optional transformation (e.g. log) to perform on the input data.
    process_data = identity) {
  map(
    aheads,
    \(ahead) {
      arx_forecaster(
        process_data(epi_data), outcome, predictors,
        args_list = arx_args_list(
          # The horizons. Number of days after the forecast date that we want to generate predictions for.
          ahead = ahead,
          # Used to derive new predictors by shifting the provided predictors backwards in time.
          # Here, raw predictors are lagged 0, 1, 2, and 3 weeks.
          lags = c(0, 7, 14, 21),
          # Train on the last 16 weeks of data.
	   n_training = as.difftime(16, units='weeks'),
          adjust_latency = "extend_lags"
        )
      )$predictions |>
        # Transform long-format dataframe to wide format with a separate column for each quantile.
        # This is just for convenience.
        pivot_quantiles_wider(.pred_distn)
    }
  ) |>
    bind_rows()
}


# Create a version of the data where we pretend that the finalized data was reported immediately and there were no revisions to the data.
nhsn_faux <- nhsn |>
  epix_as_of(nhsn$versions_end) |>
  mutate(version = time_value) |>
  as_epi_archive()

# We want forecasts for 4 weeks into the future.
aheads <- c(0, 7, 14, 21, 28)

# Generate forecasts for version faithful and un-faithful archives.
version_unfaithful <- nhsn_faux |>
  epix_slide(
    ~ forecast_wrapper(.x, aheads, "confirmed_covid_adm", "confirmed_covid_adm"),
    .before = Inf,
    .versions = forecast_dates
  ) |>
  mutate(version_faithful = "Version un-faithful")

version_faithful <- nhsn |>
  epix_slide(
    ~ forecast_wrapper(.x, aheads, "confirmed_covid_adm", "confirmed_covid_adm"),
    .before = Inf,
    .versions = forecast_dates
  ) |>
  mutate(version_faithful = "Version faithful")

forecasts <-
  bind_rows(
    version_unfaithful,
    version_faithful
)

Issues during modeling were largely due to behind-the-scenes processing, and error messages were opaque

We ran into a few errors during model training that we struggled to debug, for example,

![][/blog/2026-03-27-modeling-package-review-delphi/delphi-review-error.png]

We encountered this error multiple times, caused by automatic filtering of input data that epipredict performs behind the scenes. While the error message does provide some context, it wasn’t enough to quickly identify the underlying causes.

One instance of the error was because a subset of states stopped reporting COVID admissions, causing an automatically-selected date range to be much shorter than expected and the majority of input data to be dropped. Our solution was to remove the states that stopped being reported.

Another error instance was caused by automatic filtering keeping only those observations (rows) with no missing values in any columns. This is generally reasonable to do, except that the filter included unused (non-predictor, non-outcome) columns. Our solution in this case was to drop the unused columns before training the model.

Delphi is continually working to improve messaging and debugging experience in our packages. Feel free to reach out with specific issues or questions by email or in the project’s GitHub repository.

Evaluating predictions: The importance of revision-aware modeling

Note: This is a toy example. Our intent is to show the modeling process rather than an amazing forecast. To improve the accuracy of the model during real use, users should consider adjusting parameters, adding exogenous predictors, or doing revision prediction as a first step.

This plot demonstrates the outcome of our two models:

![Predicted distributions generated by our model][/blog/2026-03-27-modeling-package-review-delphi/delphi-review-predictions.jpg]

As before, the black line represents the finalized values of adult COVID admissions, as known in July 2025. Each solid colored line represents a version of the data as reported on a past date. The report date is indicated by the dashed vertical of the same color. Each report date is also used as a forecast date in the backtesting setup.

Lastly, the shaded fans indicate the actual forecasts made at different times based on the historical values of the same color. The central dashed line of each fan is the predicted median, and the shading indicates the 5% to 95% prediction interval.

While it is difficult to assess model performance without calculating various error metrics, it is clear that there are differences between the version faithful and unfaithful predictions.

In the plots above, we observe that the version faithful predictions made in early April (blue) start at a value of 425, while the version unfaithful predictions start around 600. This is because the data was adjusted upwards as it was finalized.

Additionally, the models can generate different trajectories, as in the February forecasts (yellow-green). While the version faithful shows a moderate drop, the version unfaithful remains relatively flat.

We can see that the version-unfaithful forecast performs better here – the predicted median is closer to the finalized data and the finalized data is entirely contained within the 5%-95% quantile fan.

However, this level of performance is misleading – it is impossible to achieve in real predictions. Remember, the version-unfaithful model is trained on finalized data that is not actually available for real prediction tasks. This is why pseudoprospective forecasting that correctly handles data revisions is so important. Without it, we have no idea of the true performance of our models.

Overview and comparison to other tools

All in all, Delphi packages provide both straightforward quick-start models and a fully customizable modeling framework (not discussed here). The quick-start models are suitable for most users and will give reasonable predictions with minimal modification.

Users only need to set a small number of parameters when using the quick-start models. These include the horizon (how far in the future to generate predictions for), and which outcome, predictors, and lags to use. Although this can be simple, expect to spend some time experimenting.

epipredict can fit a single model across many geographic locations (geopooling) or fit a different model to each location. Other packages, like Prophet, do not always have built-in functionality to deal with multiple geographic locations, which means that the user may need to implement custom handling to make geopooled models.

Unlike any other data-handling and modeling tools we’re aware of, Delphi’s packages provide utilities and support for correctly handling data with revisions. Users can also use these tools with any panel data.

Because some processing is done automatically and obfuscated from the user, the debugging process can be confusing and time-consuming, especially for those without prior statistical or coding experience in R. Compared to the error messages we see in Facebook’s Prophet package, these are less interpretable.
